## 0. この計画の使い方

この文書は、今後の開発で「次に何を直すか」を毎回考え直さないための基準です。

開発では次を守ります。

1. **1回の作業では目的を1つに絞る。**
2. **1つのPull Requestに複数の目的を混ぜない。**
3. **mainへ直接変更しない。**
4. **前段階の合格条件を満たすまで、次段階へ進まない。**
5. **ゲーム内容を増やす前に、既存処理の不具合と測定不足を直す。**
6. **数値を合わせるためだけに安全判定を緩めない。**
7. **実機未確認のものを「確認済み」と書かない。**
8. **公開用は引き続き `index.html` 1ファイルで完結させる。**

各段階は、原則として1バージョン、1ブランチ、1Pull Requestで進めます。

## 1. 完成時に守るゲームの中心

### 1-1. 一言で表すゲーム

左へ逆走する🚚を1タップのジャンプだけで操作し、夜道の穴、地上障害物、対向障害物、空中報酬を読み分けながら、遠くまで走る競技型ブラウザゲームです。

### 1-2. 変えてはいけない中心ルール

- 操作はタップまたはジャンプボタンによる1段ジャンプ
- 2段ジャンプなし
- 穴は無敵中でも即終了
- 障害物事故で警戒度が上がる
- 警戒度3で🚓に捕まり終了
- 事故後は15秒の逃走モード
- 👯‍♀️取得で障害物事故だけ4秒間無効
- 結果はkm表示
- スマートフォン縦画面を中心にする
- 大量に動くものはCanvasで扱う
- ランキング通信に失敗しても結果画面は壊さない

### 1-3. 完成時に生まれているべき判断

完成版では、単に一定間隔でジャンプするだけでなく、最低限次の判断を発生させます。

- 穴または地上障害物に対して「ジャンプする」
- 空中側の危険に対して「ジャンプしない」
- アイテムを取るか、安全を優先するか選ぶ
- 対向障害物の速度を見て、反射的にジャンプ時刻を決める
- 逃走中に、障害物・穴・アイテムの優先順位を短時間で決める
- 無敵中に障害物を突破するが、穴には注意する

# P1. 実効出現の計測

## 目的

「生成された数」ではなく、「プレイヤーが画面上で認識できた数」を測れるようにします。

## なぜ必要か

SpawnDirectorの生成数だけでは、次を区別できません。

- 画面へ出る前に消えた
- 同じ位置に重なった
- 画面端で一瞬しか見えなかった
- プレイヤーが判断できる位置まで来た
- 生成はされたが別の予定に押し出された

## 実装項目

- 各オブジェクトに `presented` を持たせる
- 認識可能なX座標を初めて通過した時だけ `presented = true`
- 次の実効カウンターを追加
  - `presentedHoleCount`
  - `presentedGroundObstacleCount`
  - `presentedOncomingCount`
  - `presentedScoreItemCount`
  - `presentedPowerupCount`
- 実効出現間隔を距離と秒の両方で記録
- 0.5km以降の最大空白時間を記録
- 同時に画面内にある強い危険数を記録
- `spawnSource` 別の実効出現数を記録

## 合格条件

- 1オブジェクトを2回数えない
- 画面へ到達しないオブジェクトを数えない
- 通常、穴間、逃走、無敵、早期対向の区別ができる
- 自動テストで生成数と実効出現数の関係が説明できる
- ゲームバランスは変更しない

## 変更禁止

- 出現間隔
- 出現確率
- 速度
- 穴幅
- 点数
- 当たり判定

# P2. 穴・障害物・アイテム密度の再調整

## 目的

SpawnDirectorを壊さず、画面上での空白と偏りを減らします。

## 調整対象

- 出現予定の進め方
- SpawnDirectorのゾーン別処理順
- 必須予定の待機方法
- 穴間障害物解決後の次穴予約
- アイテムの予定消費方法
- 同じ種類の予定が連続する時の抑制

## 暫定目標

以下は自動試験と実機で調整するための初期目標です。実機で理不尽なら数値を優先せず安全性を優先します。

### 穴

- 0〜1km: 4〜6回
- 1〜2km: 追加で6〜9回
- 2km以降の平均実効穴間隔: 0.10〜0.18km
- 通常時の最大実効穴間隔: 0.30km未満
- 穴同士の接触: 0
- 穴間障害物なし: 0

### 障害物

- 穴と穴の間に最低1回
- 同時に画面内へ出る強い地上危険は原則2個まで
- 通常時の最大判断空白: 1.2秒未満
- 1.5km以降の最大判断空白: 0.9秒未満

### 対向障害物

- 0.8km未満: 0
- 2km以内: 最低1回
- 早期最短TTC: 0.80秒以上
- 全体最短TTC: 0.50秒以上
- `unavoidableOncomingCount`: 0
- 実機では「余裕がありすぎず、反射で避けられる」速度にする

### アイテム

- 通常報酬: 1kmあたり6〜10個を初期目安
- 取り逃がし対象: 1kmあたり0〜2個を初期目安
- 👯‍♀️: 現行の出現間隔を大きく増やさない
- 同じ高さ・同じ間隔・同じ取得リズムが3回以上続く割合を低くする
- アイテム加算が実走行距離より支配的にならない

## 合格条件

- queue overflow 0
- mandatory timeout 0
- SpawnDirector整合式が成立
- 穴すり抜け0
- 2段ジャンプ必須配置0
- 地上と空中が同時に完全封鎖される配置0
- 30種類以上の固定seedで基準内
- 実機3プレイで「何もない時間が長い」と感じない

# P3. 地上・空中・穴パターン

## 目的

単体のランダム出現から、短い判断パターンへ移行します。

## パターン記号

- `G`: 地上障害物
- `O`: 対向障害物
- `A`: 空中報酬
- `H`: 穴
- `P`: 無敵
- `S`: 安全区間

## 初期パターン

- `G → S → H`
- `H → S → G`
- `G + 上空A`
- `H + 穴上A`
- `O → S → H`
- `H → S → O`
- `P → G → G`
- `G → A → H`

## 安全契約

- 同時に相反する操作を要求しない
- 着地地点へ即危険を置かない
- 地上障害物を越えた直後の穴を禁止
- 穴直後の対向障害物はTTCと着地猶予を両方満たす
- 1回のジャンプで処理できない連続配置を禁止

## 合格条件

- 10,000パターン生成で避け不能0
- 同じパターン3連続を抑制
- 単一パターンが全体の25%を超えない
- 0〜1kmでは学習用パターンだけ
- 1km以降に複合パターンを段階解禁

# P4. 空中障害物

## 目的

「ジャンプする」だけではなく、「ジャンプしない」という判断を追加します。

## 実装順

1. 空中障害物を1種類だけ試作
2. GROUND / AIR / HOLE分類へ登録
3. 地上走行なら安全、ジャンプ中は接触する高さへ配置
4. 空中障害物用の影または吊り下げ表現を追加
5. 最初は2km以降で低頻度
6. 逃走中にはまだ増やさない

## 候補

- 低い看板
- 工事用の横棒
- 垂れ下がったケーブル

既存作品の固有キャラクターや素材は使いません。

## 合格条件

- 地上にいれば必ず安全
- 通常ジャンプでは危険が分かる
- 地上障害物と同時に出て上下を完全封鎖しない
- 穴と同時に出して跳ぶ・跳ばないを矛盾させない
- 初見区間で安全に学習できる
- 1000回の配置試験で完全封鎖0

# P5. 逃走・無敵の専用構成

## 目的

逃走をゲームの山場にし、無敵を取った意味が明確に伝わるようにします。

## 逃走15秒の構成

| 経過 | 内容 |
| --- | --- |
| 0〜1秒 | 状況を理解する短い余白 |
| 1〜6秒 | 地上障害物と報酬中心 |
| 6〜12秒 | 小穴、地上、対向を混ぜる |
| 12〜15秒 | 最後の短い連続判断 |

## 1回の逃走での初期保証

- 地上障害物: 5〜8
- 穴: 2〜4
- 加点アイテム: 4〜7
- 2km以降の対向障害物: 1〜3
- 👯‍♀️は増やさない

## 無敵

- 実時間ではなく、実際にプレイしている時間で8秒進める
- バックグラウンド中に減らさない
- 無敵中に障害物を2〜3回提示
- 無敵直後に大穴を置かない
- 穴は無敵対象外のまま
- 無敵で防いだ事故を分かりやすく表示

## 合格条件

- 無敵8秒が速度倍率に影響されない
- バックグラウンド復帰で無敵だけ消えない
- 逃走中の最大判断空白0.7秒以下を目標
- 逃走中の避け不能配置0
- 逃走中でもTTC最低値を維持

# P6. スコアと競技ルールの最終固定

## 目的

ランキング実装前に、「何を競うゲームか」を確定します。

## 決定

2026-07-21にユーザー承認された**A改良版**を採用します。

```text
最終記録 = 実走行距離 + アイテム加算 - 実適用ペナルティ
ランキング p_score = 最終記録の整数メートル
```

現行式、アイテム個別点数、50%のペナルティ適用上限、ランキングRPC形状は維持します。そのうえで、アイテム種別を20個周期の💰14・🔩3・⚙️3（合計1.70km）にしてseedごとの価値偏りを抑えます。周期の開始位置と進行方向はプレイごとに変え、並び順を固定化しません。

HUDは「走行」と「記録」を分離します。結果画面とシェア文では「最終記録」を主値とし、「実走行距離」を別項目で常に確認できるようにします。

## 実装・検証項目

- `resultSnapshot.score`、結果の大表示、ランキング`p_score`を同一値にする
- 💰の取り逃がしペナルティを常に0にする
- 20個周期の種別数と加算可能総量を自動検査する
- 1000プレイ相当の固定seedストレス試験を行う
- 同距離・全取得のseed間最大幅を変更前620mから50%以上削減する
- P1〜P5、v22 UI、150km耐久を全回帰する

## 合格条件

- 1000プレイ相当でスコア分布を取得
- 同距離・全取得のseed間最大幅310m以下
- 変更前最大幅620mから50%以上削減
- 表示値と送信値が完全一致
- 仕様書、結果画面、シェア文、ランキングが同じ定義
- `P6_SCORE_COMPETITION_CONTRACT.md`と専用回帰artifactを保存

ランキングはゲーム本体が安定した後に実装します。

## 固定条件

- SupabaseのPublishable keyのみ使用
